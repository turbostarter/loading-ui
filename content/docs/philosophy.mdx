loading-ui exists to make **loading and pending states** feel deliberate: clear, accessible, and consistent with the rest of your UI — without drowning users in spinners.

## Progress over ambiguity

Prefer patterns that answer “is something happening?” at a glance: motion, placement, and copy that matches the action (submitting, saving, loading more).

## Accessible by default

Loading indicators should work with **reduced motion**, **keyboard** flows, and **screen readers** where it matters. We avoid decorative motion that communicates nothing.

## Composable and boring (in a good way)

Primitives should **compose** with your design system (tokens, typography, spacing) rather than inventing a parallel visual language. Defaults are helpful; escape hatches stay explicit.

## Documentation is part of the API

If it isn’t documented, it isn’t stable. Examples in MDX should match how real apps wire components up — including edge cases when they’re common.
